# DGIST Campus Poll

## Decision

`arabesque.cc/dgist`를 DGIST 캠퍼스 여론조사 전용 진입 랜딩으로 만들고, 그 뒤의 인테이크·엔진·큐·결과·인증·과금은 기존 앱을 그대로 재사용한다.

- `/dgist`는 **랜딩 한 장만 신규**다. 별도 미니앱을 만들지 않는다.
- 페르소나 풀은 이미 등록된 `dgist` 풀(2,895명)을 쓴다. 풀 등록 코드는 이미 존재하므로 parquet 파일만 연결한다.
- 여론조사 유형 10종 중 9종은 기존 시뮬레이션 엔진의 **프론트 라벨만 캠퍼스용으로 교체**한다. 백엔드 `simulation_type` 키는 바꾸지 않는다.
- `campus_policy`(정책 찬반) 1종만 신규로 만든다. 기존 10종 중 찬성/반대 비율을 직접 산출하는 것이 없기 때문이다.
- 자유 입력은 기존 히어로 입력창 + 인테이크 라우팅을 그대로 쓴다.
- 브랜딩은 minsim을 유지하고 카피만 캠퍼스용으로 바꾼다. DGIST 로고·색상은 쓰지 않는다(학교 승인 없는 공식 사칭 방지).

## Context

### 이미 존재하는 것

| 자산 | 위치 | 상태 |
| --- | --- | --- |
| `dgist` 페르소나 풀 등록 | `src/data/pools.py:34` | 등록됨, parquet 부재로 `available: false` |
| DGIST 페르소나 parquet | `koresim-v2/data/dgist_personas.parquet` | 2,895행 15열, Nemotron 필드 완전 호환 |
| JSON → parquet 변환기 | `scripts/import_dgist_personas.py` | 완성됨 (이번 범위에서는 미사용) |
| 풀 선택 UI | `frontend/src/v2/MinsimIntakeFlow.tsx:405` | `/api/config`의 `persona_pools`를 읽어 자동 활성화 |
| 풀 요청 파싱 | `frontend/src/intake/payloadBuilder.ts:280` | `dgist` 값 통과 |
| 자유 입력 → 인테이크 | `frontend/src/LandingPage.tsx` `start()` | `sessionStorage['minsim.heroPrompt']` → `/projects` |

### 페르소나 풀 구성 (실측)

- 총 2,895명. 남 1,766 / 여 1,129.
- 학력: 학사 재학 1,096 · 석사 재학 523 · 박사 328 · 석박통합 재학 209 · 박사 재학 203 · 석사 190 · 박사후연구원 111 · 전문학사 90.
- 직군: DGIST 기초학부 학부생(전공 관심별 세분) 중심 + 대학원 연구원 + 행정/교직원.
- 거주: 현풍 원룸 686 · 기숙사/원룸 혼합 543 · 기숙사(비슬빌리지) 532 · 대구 시내 가족 주거지 497 · 교직원 관사 170.
- 서사 필드에 E1·R1 식당, E7 카페, 현풍·테크노폴리스 생활권이 실제로 기술되어 있어 학식·셔틀·기숙사 안건에 근거가 있다.

`province`/`district`는 **출신지 또는 가족 주거지**이지 소속이 아니다. 전원 DGIST 소속이므로 "대구 21%"를 소속 분포로 읽으면 안 된다.

그러나 이 필드는 **귀향 비용의 프록시**로서 의사결정에 직접 쓰인다. 근거리(대구·경북) 출신은 주말에 본가로 가고 원거리 출신은 캠퍼스에 남으므로, 주말 잔류 인구가 이 분포로 결정된다. 축제 요일(금 vs 토), 도서관 심야·주말 수요, 셔틀·귀향버스 노선은 출신지 분포 없이는 정할 수 없다. 지역 분해는 유지하되 **해석 프레임을 "소속"이 아니라 "귀향 비용"으로 명시**한다.

## Architecture

```
arabesque.cc/                기존 랜딩 — 상단 네비 + 하단 CTA에서 /dgist 진입
arabesque.cc/dgist           신규 캠퍼스 랜딩 (공개, 로그인 불필요)
      ↓  자유 입력 또는 유형 카드 선택
      ↓  sessionStorage: minsim.heroPrompt, minsim.personaPool = 'dgist'
arabesque.cc/app             기존 프로젝트 흐름 (풀이 DGIST로 선점된 상태)
      → /projects/:id/type → /projects/:id/intake → /loading → /results
```

경계는 하나다. `/dgist` 랜딩은 **의도(질문 + 유형 + 풀)를 sessionStorage에 심고 기존 흐름으로 넘기는 것**이 전부다. 랜딩은 API를 호출하지 않고, 실행 상태를 갖지 않는다. 따라서 기존 앱을 개선하면 DGIST 경로에도 자동 반영된다.

## Components

### 1. 페르소나 풀 연결

`data/dgist_personas.parquet` → `koresim-v2/data/dgist_personas.parquet` 심볼릭 링크.

`pool_metadata()`가 `spec.path.exists()`만 검사하므로 링크 생성만으로 `/api/config`가 `available: true`를 반환하고 인테이크 풀 선택기가 활성화된다. **파이썬 코드 변경 없음.**

### 2. `campus_policy` 시뮬레이션 (신규)

기존 10종 중 단일 안건에 대한 찬성/반대 비율을 산출하는 것이 없다. `churn_prediction`이 유지/관망/이탈 3분할로 구조는 유사하나 프롬프트가 고객 이탈 맥락으로 작성되어 있어 재사용하면 응답 품질이 떨어진다.

**입력 스키마**

| 필드 | 필수 | 설명 |
| --- | --- | --- |
| `agenda` | 필수 | 안건 한 줄 (예: 중앙도서관 24시간 개방) |
| `current_state` | 필수 | 현행 상태 (예: 평일 09–23시, 주말 10–18시) |
| `proposed_change` | 필수 | 변경 내용 |
| `tradeoffs` | 선택 | 예상 부작용·비용 (예: 연 운영비 증가, 야간 인력 필요) |

`tradeoffs`가 비면 응답이 전부 찬성으로 쏠린다. 선택 필드지만 인테이크에서 강하게 유도한다.

**페르소나 응답 스키마**

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| `stance` | enum | `찬성` · `조건부찬성` · `반대` · `판단유보` |
| `reason` | string | 본인 생활 맥락에 근거한 이유 |
| `condition` | string \| null | `조건부찬성`일 때만 채워지는 전제 조건 |
| `intensity` | int 1–5 | 입장의 강도 |

`intensity`를 두는 이유: 여론조사 의사결정에서 "찬성 60%"보다 "강하게 반대하는 20%"가 실제 집행 리스크를 좌우한다. 강도 없는 비율은 반대 여론의 폭발력을 감춘다.

**집계 출력**

- 입장 분포 % (4분할)
- 강도 가중 찬성률 (`intensity`로 가중한 순찬성)
- 계층 교차: 학부생 / 석·박사 재학 / 박사후연구원 / 교직원
- 거주 교차: 기숙사 / 현풍 원룸 / 대구 시내 통근
- 조건부 찬성의 조건 클러스터 (무엇을 보장하면 찬성으로 넘어오는가)
- 반대 사유 top-N

**파일**

- `src/simulations/campus_policy.py` — 프롬프트, 응답 파서, 집계
- `src/simulations/registry.py` — `campus_policy` 스펙 등록
- API 결과 스키마 + 프론트 TypeScript 타입 (기존 parity 테스트 대상)
- `frontend/src/v2` 결과 렌더러 등록

### 3. 캠퍼스 유형 사전

`frontend/src/dgist/campusSimulations.ts` — 캠퍼스 라벨과 백엔드 `simulation_type` 키의 매핑 테이블 하나. 백엔드 키는 절대 바꾸지 않는다.

기본 노출 6종:

| 캠퍼스 유형 | 예시 질문 | 엔진 |
| --- | --- | --- |
| 정책 찬반 | 중앙도서관 24시간 개방, 기숙사 통금 폐지 | `campus_policy` (신규) |
| 행사·라인업 반응 | 축제 헤드라이너 A/B/C안, 초청 연사 후보 | `creative_testing` |
| 요금 변경 | 학식 500원 인상 + 반찬 추가, 기숙사비, 셔틀 유료화 | `price_optimization` |
| 신규 서비스 수요 | 심야 셔틀, 무인 편의점, 24시 스터디카페 | `product_launch` |
| 예산·시설 우선순위 | 헬스장 vs 스터디룸 vs 카페 vs 주차장 | `competitive_positioning` |
| 만족도 진단 | 현 학식·상담센터·셔틀 만족도, 학교 이미지 | `brand_perception` |

더보기 4종:

| 캠퍼스 유형 | 예시 질문 | 엔진 |
| --- | --- | --- |
| 공지 채널 전략 | 에브리타임 vs 카톡 vs 메일 vs 포스터 | `campaign_strategy` |
| 의견 갈림 지점 | 이 안건에서 어떤 집단이 어떻게 갈리나 | `market_segmentation` |
| 이탈·불만 위험 | 자퇴·휴학 고민, 동아리·프로그램 중도포기 | `churn_prediction` |
| 문구 설득력 비교 | 총학 공약 문구 A/B, 캠페인 카피 | `value_proposition` |

**직접 입력**: 기존 히어로 자유 입력창 → `minsim.heroPrompt` → 인테이크 플래너가 유형을 자동 라우팅. 신규 구현 없음.

### 4. `/dgist` 랜딩

`frontend/src/dgist/DgistLandingPage.tsx` — 기존 `LandingPage.tsx`의 섹션 구조를 미러링하되 카피와 예시만 캠퍼스용.

- 히어로: "DGIST에 무엇을 물어볼까요?" / "질문 한 줄을 적으면 DGIST 구성원 2,895명 합성 패널이 답합니다."
- 예시 칩: 도서관 24시간 개방 / 학식 500원 인상 / 축제 헤드라이너 A vs B / 심야 셔틀 신설
- 유형 그리드: 6종 + 더보기 4종
- 표본 배지: 학부 1,096 · 석사 재학 523 · 석박통합 209 · 박사 328 · 박사후 111 · 교직원 · 기숙사 거주 약 51%
- 면책: 기존 문구 유지. 합성 페르소나 시뮬레이션이며 실제 설문조사가 아님을 명시한다.

라우팅: `frontend/src/v2/navigation.ts`의 `parseV2Route`에 `/dgist` → `{ page: 'dgist-landing' }` 추가, `Root.tsx`에서 분기.

### 5. 계층·거주 분해 패널 (지역 분해에 **추가**)

두 축은 서로 다른 질문에 답하므로 **둘 다 렌더링한다.** 지역 분해를 제거하지 않는다.

**계층 × 거주형태 교차 히트맵 (신규)** — "누가 이 정책의 수혜자·부담자인가"

- 행: 학부생 / 석·박사 재학 / 박사후연구원 / 교직원
- 열: 기숙사 / 현풍 원룸 / 대구 시내 통근
- 셀: 강도 가중 순찬성 %p + 표본 수

**출신지 분해 (기존 유지, 해석 프레임 변경)** — "주말·심야에 캠퍼스에 몇 명이 남는가"

`KoreaReactionMap`의 지역 데이터를 그대로 쓰되, 라벨과 설명을 **귀향 비용 프록시**로 바꾼다. `province`를 소속으로 읽지 않도록 패널 상단에 해석 지침을 고정 노출한다.

두 패널 모두 `segment-reaction-radar`의 신뢰 규칙을 승계한다. 높은 비율과 높은 신뢰도를 같은 의미로 취급하지 않고, 표본 수를 항상 수치 옆에 표시하며, 표본 20 미만 셀은 숨기지 않고 비교 불가로 표시한다. 정렬은 최고 비율이 아니라 신뢰 우선이다.

임계값 규칙은 **표시와 반드시 일치해야 한다.** 목업 검토에서 "표본 20 미만은 흐리게"라고 적어놓고 `n=12`, `n=14` 셀을 진하게 그린 오류가 발견됐다. 신뢰도 표기가 자기 규칙과 어긋나면 표기 전체의 신뢰가 무너지므로, 임계값은 상수 하나로 두고 렌더러와 문구가 같은 상수를 참조한다.

분기 기준은 `persona_pool === 'dgist'`이며, 기존 nationwide 결과 화면은 변경하지 않는다.

### 5b. 인테이크 입력 UX

자유 질문이 1급 입력이고, 구조화 필드는 AI가 채우는 초안이다.

1. 상단에 자유 질문 박스: "DGIST 학생들에게 묻고 싶은 것, 여론조사해보고 싶은 것을 물어보세요"
2. `agenda`(안건)만 필수. `current_state` / `proposed_change` / `tradeoffs`는 AI가 채운다.
3. 칸마다 `AI로 채우기` · `AI로 다시 뽑기`, 하단에 `전체를 AI로 다듬기`(안건 기준 3칸 일괄 재생성).
4. AI가 채운 칸은 **`AI 초안 · 사실 확인 필요` 마크**를 단다. `되돌리기` / `직접 수정` 제공.
5. 빈 칸은 회색 placeholder로 예시 문장을 보여준다.

4번이 필수인 이유: `current_state`(도서관 운영시간, 현행 학식 가격)는 LLM이 **모르는 사실**이다. 그럴듯하게 지어내며, 틀린 현행 상태로 조사를 돌리면 결과 전체가 무의미해진다. 마크 없이 채우면 사용자가 검증 없이 실행한다.

### 5c. 계층 최소 표본 보장 (층화 추출)

현재 `PersonaSampler`는 단순 무작위다. 2,895명 중 교직원 비중이 낮아 200명 표본에서 교직원이 3~5명만 뽑힌다. **계층별 비교가 성립하지 않는다.**

`campus_policy`와 DGIST 풀 조합에서는 계층(학부생 / 석·박사 재학 / 박사후연구원 / 교직원) 최소 표본을 보장하는 층화 추출을 적용한다.

- 요청 표본이 계층 최소치 합보다 작으면 오류가 아니라 최소치를 비례 축소하고 경고를 낸다.
- 층화로 인해 원 모집단 비율이 왜곡되므로, **전체 집계는 모집단 비율로 역가중**한다. 계층별 수치는 비가중 원값을 쓴다.
- 결과 메타데이터에 `sampling: stratified` 와 계층별 실제 표본 수를 기록한다.

역가중을 하지 않으면 "교직원을 일부러 많이 뽑았기 때문에 반대가 높게 나온" 결과가 전체 찬성률로 보고된다.

### 5d. 실제 설문 문항 생성

시뮬레이션 결과에서 **진짜 설문 문항**을 생성한다. 시뮬은 사전 탐색이고 실제 설문이 검증이므로, 이 다리가 없으면 합성 결과가 그대로 의사결정 근거로 쓰이는 위험이 남는다.

- 입력: `campus_policy` 결과 (조건 클러스터, 반대 사유, 계층 격차)
- 출력: 찬반 문항 1개 + 조건 클러스터 기반 다지선다 + 반대 사유 우선순위 문항 + 자유서술 1개
- 형식: 마크다운 및 구글폼 붙여넣기용 평문
- 결과 화면에서 복사 가능. 외부 공유 전 사람 검토는 기존 export 정책을 따른다.

### 5e. 반대파 설득 시뮬

반대(`stance` = 반대)한 페르소나에게만 조건을 제시하고 재질문해 전환율을 산출한다. 기존 `followup_service.run_followup`의 코호트 선택을 재사용한다.

- 입력: 조건 문장 1개 (예: 야간 경비를 외주로 운영, 교직원 당직 없음)
- 출력: 반대 → 찬성/조건부찬성 전환율, 전환된 페르소나의 이유, 끝까지 반대한 사유
- 조건 클러스터의 자동화 버전이며, 1:1 인터뷰를 코호트 전체에 한 번에 돌리는 것에 해당한다.

  ├─ 자유 입력 → sessionStorage['minsim.heroPrompt']
  ├─ 유형 카드 → sessionStorage['minsim.simulationType'] (백엔드 키)

       ↓ 풀 선택기가 sessionStorage 값을 기본값으로 채택

결과 envelope → MinsimResultsPage
       └─ persona_pool === 'dgist' 이면 계층·거주 히트맵 추가 +

## Error Handling

- **parquet 부재**: `pool_metadata()`가 `available: false`를 반환하고 풀 선택기가 비활성화된다. 기존 동작이며 크래시하지 않는다. `/dgist` 랜딩은 `/api/config`를 조회해 `dgist` 풀이 비활성이면 실행 CTA를 비활성화하고 사유를 표시한다.
- **`campus_policy` 응답 파싱 실패**: 기존 partial result 정책을 따른다. 실패 건은 `parse_failed`로 집계하고 나머지로 결과를 낸다.
- **`stance` enum 이탈**: 파서가 알 수 없는 값을 `판단유보`로 강등하지 않고 파싱 실패로 처리한다. 조용한 강등은 찬반 비율을 왜곡한다.
- **교차 셀 표본 부족**: 셀을 숨기지 않고 비교 불가 상태로 표시한다.
- **인증**: 변경 없음. `/dgist` 랜딩은 공개, 실행은 기존 Google OAuth + 무료 5회 usage ledger를 그대로 적용받는다.

## Known Defects (2026-07-20 라이브 200명 검증에서 발견)

증거: `docs/verification/dgist-campus-poll-v1-live.json` (Upstage solar-pro2, 200명, seed 42, parse_failed 0)

정상 동작한 것: 층화 추출(학부 80 / 석박 58 / 박사후 23 / 교직원 39), 파싱 0 실패, 출신지 분해, 저신뢰 셀 표시.

**D-1. Stance collapse — 찬반 분포가 무의미하다.**
200명 중 조건부찬성 190, 반대 10, 찬성 0, 판단유보 0. 4분할이 사실상 1분할이다.
현재 프롬프트에서 `조건부찬성`이 항상 가장 안전한 답이므로 모델이 전부 여기로 몰린다.
`net_support 32.2%p`는 "거의 모두가 조건부"라는 사실을 숫자 하나로 덮는다.

**D-2. 조건 클러스터가 클러스터되지 않는다.**
조건부찬성 190명에서 서로 다른 조건 문자열이 178개 나왔다. 최다 빈도가 4다.
설계에서 이 클러스터를 "이 안건의 실질 결정권"으로 놓았는데, 178행짜리 유사 문장 목록은
의사결정에 쓸 수 없다. 자유 서술 `condition`을 문자열 그대로 `Counter`에 넣은 것이 원인이다.

**D-3. 반대 사유도 동일하게 클러스터되지 않는다.** 10건 모두 빈도 1이다.

**D-4. 계층×거주 교차에 신호가 없다.** 모든 셀이 +21~+40 사이다. D-1의 직접적 결과다.

### 수정 방향 (V1.1)

1. 프롬프트에서 `조건부찬성`을 기본 선택지가 아니라 **예외**로 규정한다. "조건 없이 지금 안대로
   시행해도 괜찮으면 찬성, 어떤 조건으로도 반대면 반대"를 먼저 판단하게 하고, 조건부는 명시적
   전제가 있을 때만 허용한다.
2. `condition`을 자유 서술 단독이 아니라 **고정 taxonomy 선택 + 자유 서술 보조**로 바꾼다.
   안건별 taxonomy는 인테이크에서 생성하거나 `tradeoffs`에서 파생한다.
3. `opposition_reasons`도 같은 방식으로 카테고리화한다.
4. 카테고리화 전까지 결과 화면은 "조건 클러스터"를 대표 지표로 노출하지 않는다.

D-1이 해결되기 전에는 이 시뮬레이션을 실제 의사결정 근거로 홍보하지 않는다.

## D-1 원인 진단 (2026-07-20, 라이브 실험 6건)

각 10~20명, Upstage solar-pro2, seed 42, 모든 실험 parse_failed=0.

### 실험 1: 입력이 원인인가 → 아니다

`tradeoffs`를 바꿔가며 stance 분포를 관찰했다.

| 입력 | 결과 |
| --- | --- |
| 미확정 택일 (원본) | 조건부찬성 10 / 10 |
| 확정 재원, 학생 부담 없음 | 조건부찬성 10 / 10 |
| 확정 부담, 학생회비 연 2만원 인상 | 조건부찬성 10 / 10 |

학생이 직접 돈을 내는 조건에서도 100% 조건부찬성이다. **입력 결함이 아니다.**

### 실험 2: 프롬프트 지시로 고쳐지는가 → 아니다

2단계 판단 절차와 "대부분은 찬성 또는 반대에 속한다, 조건부찬성은 예외다"를 명시했다.

| 프롬프트 | 결과 |
| --- | --- |
| 현행 | 찬성 1 / 조건부 9 |
| 2단계 판단 강제 | 찬성 0 / 조건부 10 |
| 2단계 + 개인 맥락 강조 | 찬성 1 / 조건부 9 |

명시적 지시를 모델이 무시한다. **프롬프트 문구 조정으로는 해결되지 않는다.**

### 실험 3: 스키마 구조를 바꾸면 → 해결된다

| 스키마 | 결과 (n=20) |
| --- | --- |
| `stance ∈ {찬성, 반대}`, 조건 첨부 불가 명시 | **반대 16 / 찬성 4** |
| `support ∈ 0~10` 수치 척도 | 7이 11명, 8이 6명 (7~8에 재쏠림) |

**결론: 헤지 선택지가 스키마에 존재하는 한 모델은 항상 그리로 간다.**
프롬프트가 아니라 열거형에서 `조건부찬성`을 제거해야 한다.

수치 척도도 대안이 되지 못한다. 7~8에 다시 몰리며, "7점"은 그 사람이 실제 투표에서
어느 쪽에 표를 던질지 알려주지 않는다. 여론조사는 이분 선택이 의사결정 형태와 일치한다.

### 수정 확정 (V1.1)

1. `stance`를 `찬성` / `반대` 2분할로 축소한다. `조건부찬성`을 열거형에서 제거한다.
2. `판단유보`는 "안건 자체를 이해할 수 없을 때"로 좁게 정의해 유지한다.
   (이 선택지가 새 헤지 통로가 되는지는 아직 검증하지 않았다. 구현 시 재측정한다.)
3. `condition`은 stance와 무관하게 모든 응답에서 선택 수집한다.
   "이것만 보장되면 마음이 바뀐다"를 찬성자에게도 묻는 것이 D-2 taxonomy의 입력이 된다.
4. 프롬프트에 "조건을 붙일 수 없다, 지금 안 그대로 시행된다"를 명시한다.

## campus_priority (순위형) — 2026-07-20 추가

모든 여론조사가 찬반형이 아니다. "복지예산을 어디에 먼저 쓸까"처럼 순위를 묻는
질문은 기존 11종 어디에도 맞지 않았다. 배분(몇 대 몇)은 실무에서 순위로 근사할
수 있으므로 순위 하나로 커버한다.

- 입력: `question` + `items`(3~6개) + `context`
- 응답: `ranking`(전체 순위) + `top_reason` + `bottom_reason`
- 집계: Borda count, 평균 순위, 1위 지목률, 계층별 순위, **계층 간 순위 역전**

Borda를 쓰는 이유: 1위 지목률만 보면 2~3위 선호가 통째로 버려지고, 평균 순위만
보면 극단적 호불호가 평균에 묻힌다.

### 라이브 검증 (30명, Upstage solar-pro2, seed 42, parse_failed 0)

| 순위 | 항목 | Borda | 평균 | 1위 지목 |
| --- | --- | --- | --- | --- |
| 1 | 학식 질 개선 | 84 | 1.20 | 93.3% |
| 2 | 심야 셔틀 신설 | 52 | 2.27 | 6.7% |
| 3 | 헬스장 확충 | 26 | 3.13 | 0.0% |
| 4 | 스터디룸 증설 | 18 | 3.40 | 0.0% |

순위 역전 2건 관측:
- 심야 셔틀 신설 — 박사후연구원 1위 ↔ 교직원 3위

## 민감도 재측정 — 층화 추출 100명 × 3변형 (2026-07-20)

계층별 표본 학부 31 / 석·박사 31 / 박사후 15 / 교직원 23.

| 변형 | 전체 찬성 | 학부생 | 석·박사 | 박사후 | 교직원 |
| --- | --- | --- | --- | --- | --- |
| V1 "찬성 또는 반대" | 50.0% | 74.2 | 41.9 | 33.3 | 39.1 |
| V2 "반대 또는 찬성" | 25.3% | 48.4 | 22.6 | 0.0 | 13.6 |
| V3 부정 프레이밍 | 35.0% | 41.9 | 54.8 | 0.0 | 21.7 |

| 지표 | n=20 (계층별 4~8) | **n=100 (계층별 15~31)** |
| --- | --- | --- |
| 절대 찬성률 폭 | 30.0%p | **24.7%p** |

| tau V2↔V3 | +1.00 | **+0.67** |

### 결론 — 앞선 "반증" 서술을 다시 정정한다

n=20에서 관측한 tau 음수는 **얇은 표본의 노이즈**였다. 계층별 표본을 15 이상으로
올리자 tau가 +0.67~+1.00으로 뒤집혔다. `campus_priority`에서 겪은 것과 같은 함정을
같은 방식으로 확인했다.

**확정 사항**

1. **절대 찬성률은 쓸 수 없다.** 폭 24.7%p. 선택지 순서만 바꿔도 50.0% → 25.3%.
2. **계층 간 상대 순위는 상당히 안정적이다.** 학부생이 3변형 모두 상위, 박사후연구원이
   3변형 모두 최하위권. V3에서만 석·박사가 학부생을 앞서 tau가 0.67로 내려간다.

**제품 방침**

- `campus_policy`는 **"몇 %가 찬성하는가"가 아니라 "어느 계층이 상대적으로 더 찬성하는가"**를
  판다. 절대 찬성률은 화면에서 보조 지표로 내리고, 계층 비교를 1급으로 올린다.
- 계층 비교의 신뢰 조건은 **계층별 표본 15 이상**이다. 층화 추출이 이미 이를 보장한다.
  `LOW_CONFIDENCE_MIN_SAMPLE = 20`은 이 실측과 정합적이다.
- 절대 수치를 제시해야 하는 경우 **"프롬프트 문구에 따라 ±25%p 변동"**을 함께 밝힌다.

## 회전 배치 회귀 — 원인 규명 (2026-07-20)

**파서 문제가 아니었다.** 회전된 4가지 순서 전부 파서가 정상 수용한다(무료 검증 완료).

**실제 원인: `BatchSimulator.run()`은 클라이언트를 소유할 때 1회용이다.**

`src/agent/simulator.py`
- `__init__`: `self._owns_client = llm_client is None` → 직접 만든 클라이언트를 소유
- `run()` 종료부(129~130행): 소유한 클라이언트를 `close()` 한다

따라서 **같은 인스턴스로 `run()`을 두 번 호출하면 두 번째부터 전부 실패**한다.
에러 메시지는 `upstage connection failed.`

### 실측

| 구성 | batch0 | batch1 |
| --- | --- | --- |
| 인스턴스 1개 재사용 | ok 10/10 | **err 10/10** |
| 배치마다 새 인스턴스 | ok 10/10 | **ok 10/10** |

회전 배치 코드가 하나의 simulator로 4번 `run()`을 호출했기 때문에 offset≠0인 3그룹,
즉 75/100이 실패했다. 회전 로직 자체는 옳았다.

### 이건 campus_priority만의 문제가 아니다

`BatchSimulator`는 `run()` 이후 조용히 사용 불능이 된다. 두 번째 호출이 예외 대신
**전건 실패 결과**를 돌려주므로, 호출자가 파싱 실패로 오해하게 된다. 실제로 그렇게
오진했다.

### 다시 구현할 때

1. **배치마다 새 `BatchSimulator` 인스턴스를 만든다.** (검증 완료: 10/10 · 10/10)
2. 또는 `BatchSimulator`에 다회 호출 지원을 넣는다. 이 경우 닫힌 클라이언트로
   재호출 시 **명확한 예외**를 던지게 해서 전건 실패로 위장되지 않게 한다.
3. stub LLM으로 2회 연속 `run()` 회귀 테스트를 먼저 세운다. 이번에 게이트가 못 잡은
   지점이 정확히 여기다.

## 회전 배치 재구현 (2026-07-21)

### 1차 — 파싱 회귀 수정 (`d3ff64b`)

`BatchSimulator`가 소유 클라이언트를 `run()` 후 닫는다는 근본 원인에 맞춰,
배치마다 새 인스턴스를 만들도록 고쳤다. 재호출 시 전건 실패로 위장되지 않도록
`RuntimeError`를 던지는 가드와 회귀 테스트 3건(`tests/test_batch_simulator_lifecycle.py`)을
추가했다. 라이브 20명 `parse_failed=0`.

### 2차 — 회전 기준을 정규 순서로

1차 후 100명 × 3변형으로 재측정했더니 **tau가 개선되지 않았다**(V1↔V2 `+0.33` 그대로).

원인: 회전 기준이 **입력 순서**였다. 항목 4개의 순열 24개 중 4개씩만 쓰는데,
입력이 `[A,B,C,D]`면 `{ABCD, BCDA, CDAB, DABC}`, `[D,C,B,A]`면 `{DCBA, CBAD, BADC, ADCB}`로
두 집합이 전혀 겹치지 않는다. 그래서 상쇄가 일어나지 않았다.

`sorted(items)`를 회전 기준으로 바꿨다. 입력 순서와 무관하게 같은 회전 집합을 쓴다.
`test_rotation_set_is_independent_of_input_order`로 고정했다.

### 미검증

**정규 순서 적용 후의 tau를 실측하지 않았다.** 메커니즘(회전 집합 일치)은 유닛
테스트로 확인했으나, 그것이 실제로 tau를 올리는지는 100명 × 3변형 = 300콜 재측정이
필요하다. 그때까지 `campus_priority`의 전체 1위는 여전히 단정하지 않는다.

## 정규 회전 검증 (2026-07-21, 층화 100명 × 3변형, parse_failed 0·0·0)

| --- | --- |
| V1 원순서 | 기숙사 › 대학원 › 학부실험실 › 주차장 |
| V2 역순서 | 기숙사 › 대학원 › 학부실험실 › 주차장 |

### 해석 — 정확히 무엇이 해결됐나

정규 회전은 `sorted(items)`를 기준으로 쓰므로 **사용자 입력 순서를 결과에서 완전히
제거한다.** 그래서 V1/V2/V3가 동일한 실행이 되고 tau가 1.00이 된다.

이것이 원래 발견한 문제(항목 순서를 뒤집자 1위가 학부실험실 → 기숙사로 이동)의
직접적 해결이다. 이제 사용자가 항목을 어떤 순서로 입력하든 같은 결과를 얻는다.

**주의**: tau 1.00은 "동일 실행"의 결과이므로, 위치 편향 자체가 사라졌다는 증거는
아니다. 각 페르소나가 서로 다른 회전을 보므로 위치 편향은 항목 간에 균등 분산되지만,
그 균등화의 효과 크기는 별도로 측정하지 않았다.

**부수 관측**: 회전 전 원순서에서 1위였던 "학부 실험실 장비 교체"가 정규 회전 후
3위로 내려갔다. 회전 전 1위는 **첫 번째로 제시된 위치 이점**이었을 가능성이 크다.

### 제품 방침 갱신

`campus_priority`는 이제 **전체 1위를 말할 수 있다.** 입력 순서에 좌우되지 않는다.

## D-1 최종 해결 — 찬반/조건 분리 (2026-07-21)

### 발견: taxonomy 적용 후 오히려 악화됐다

C-4 재측정(200명)에서 `dominant_stance = 조건부찬성 **100.0%**`, 계층 셀 폭 **2.7%p**.
tau +0.67~1.00을 측정한 실험은 **2지/3지 강제 프롬프트**였고, 조건부찬성이 있는
프로덕션 4지에서는 전원이 그리로 몰려 계층 비교 자체가 성립하지 않았다.

즉 화면에 "계층 × 거주형태 — 이 결과의 핵심"이라고 올려둔 상태에서, 실제로 돌리면
그 표가 전부 같은 값이었다. **화면이 거짓말을 하고 있었다.**

### 수정: stance 2지 + 조건 전원 수집

- `STANCES`에서 `조건부찬성` 제거 → `찬성` / `반대` / `판단유보`
- 프롬프트를 2단계 판단으로: ① 조건 없이 그대로면 찬반? ② 별개로 무엇이 바뀌면?
- `condition`을 stance와 무관하게 **전원에게서** 수집 (찬성자의 조건도 실행 조건이다)
